The Machine Learning Workflow

Dr. I.M. Simeon, ABE Dept., FUTMINNA · Compiled from GET 307 materials
Section 1

What Is an ML Workflow?

A machine learning workflow is a structured, step-by-step process for developing, training, evaluating, and deploying machine learning models. It organizes the entire lifecycle of a model — from raw data collection and preparation to a validated, deployable solution.

The ML workflow is also referred to as an ML pipeline, modeling lifecycle, or end-to-end machine learning process. By following a defined workflow, ML projects become more reproducible (same steps → same results), interpretable (clear documentation of each decision), and efficient (no wasted effort, clear responsibilities).

🗞️ Analogy: Building a House

You wouldn't start laying bricks before drawing architectural plans, surveying the land, and laying the foundation. Similarly, you cannot train a model before understanding your problem, collecting quality data, and preparing it properly. Skipping steps in ML leads to the same disaster as skipping steps in construction: a structure that looks finished but will collapse under real-world conditions.

Section 2

The 10 Core Stages of the ML Workflow

  1. Problem Definition

    Identifying the goal, business objective, and key performance metrics before touching any data. Key questions: What decision will this model support? What does "success" look like (accuracy? revenue? lives saved)? Is ML the right tool at all?
    Output: A clear problem statement and success criteria.

  2. Data Preparation

    Initial processing of raw data to make it usable. Includes: data collection (sourcing from databases, sensors, APIs, scraping), data cleaning (handling missing values, outliers, duplicate records, formatting errors), and basic transformations. This is the most time-consuming stage — data scientists spend roughly 60–80% of project time here.

  3. Exploratory Data Analysis (EDA)

    Systematic exploration of the dataset to understand its structure and behavior. Involves: summary statistics (mean, median, std deviation), visualisations (histograms, scatter plots, box plots, heatmaps), and correlation analysis to identify patterns, trends, anomalies, and relationships between variables.
    Goal: Form hypotheses and guide feature engineering decisions.

  4. Feature Engineering

    Creation, selection, and transformation of input variables (features) to improve model performance. Techniques include: normalisation/standardisation (scaling features to comparable ranges), encoding (converting categorical variables like "city name" into numbers), interaction terms (e.g., multiplying two features to capture their combined effect), and feature removal (eliminating irrelevant or redundant variables that add noise, not signal).

  5. Data Splitting

    Partitioning the dataset into separate subsets to ensure unbiased assessment. The standard approach divides into three sets:
    → Training set (typically 70–80%): Used to fit the model.
    → Validation set (optional, ~10–15%): Used to tune hyperparameters and select models.
    → Test set (typically 15–20%): Held out entirely; only used once at the end for final evaluation.
    Critical rule: the test set must NEVER be seen by the model during training.

  6. Training Data (Deep Dive)

    The training subset is used exclusively to fit the model. The algorithm adjusts its internal parameters to minimize prediction error on this data. What the model "knows" comes entirely from this data — which is why data quality matters so much. Garbage in, garbage out.

  7. Test Data (Deep Dive)

    An independent subset reserved for evaluating the trained model. It is not seen by the model during training and provides an honest estimate of how the model will perform on truly new, real-world data. Using the test set for training or tuning (data leakage) leads to overly optimistic performance estimates that won't hold in deployment.

  8. Model Training

    Application of a machine learning algorithm to the training data. The model adjusts its parameters to minimize error or maximize performance based on a defined objective function (also called loss function or cost function).
    For regression: minimize mean squared error. For classification: minimize cross-entropy loss. The optimizer (e.g., gradient descent) iteratively updates parameters in the direction that reduces the loss.

  9. Model Evaluation

    Quantitative assessment of model performance using the test data. Common metrics include:
    → Accuracy, Precision, Recall, F1-score (for classification problems)
    → RMSE, MAE, R² (for regression problems)
    → ROC-AUC (for binary classification)
    See Topic V for full explanations of each metric.

  10. Model Improvement

    Iterative refinement based on evaluation results. May involve: hyperparameter tuning (adjusting learning rate, epochs, batch size), feature re-engineering (adding or removing features), trying alternative algorithms, or collecting additional data. ML development is inherently cyclical — you loop through Stages 2–10 multiple times.

"By following a defined workflow, ML projects become more reproducible, interpretable, and efficient."

Section 3

Understanding the Training / Validation / Test Split

A critical concept that GET 307 students must deeply understand is why we split data, and what each split is for. This is analogous to the academic process:

🗞️ Analogy: The Exam System

Training data = Your textbook and lecture notes. This is where you learn everything. You study it repeatedly.

Validation data = Practice exams. You use these to check your understanding and adjust your study strategy (which chapters to focus on).

Test data = The actual exam. You see it only once, at the end. Your score here is the honest, unbiased measure of what you actually learned. If you "studied" the exam questions beforehand (data leakage), your score tells us nothing about your real knowledge.

Section 4

Regression vs. Classification: The Fundamental Distinction

Within supervised learning, every problem is either a regression problem or a classification problem. Understanding this distinction is fundamental before learning specific algorithms.

Regression Classification
Output Type Continuous number Discrete category / class
Core Question "How much?" or "How many?" "Which class does this belong to?"
Engineering Examples Crop yield (kg/ha), fuel consumption (L/100km), temperature forecast (°C) Equipment failure (yes/no), soil type (clay/loam/sand), crop disease type
Evaluation Metrics MSE, MAE, RMSE, R² Accuracy, Precision, Recall, F1, ROC-AUC
📖 Key Formula: Mean Squared Error (MSE)

MSE = (1/n) × Σ(yᵢ − ŷᵢ)²

Where yᵢ is the actual value, ŷᵢ is the predicted value, and n is the number of observations. MSE penalises large errors heavily because differences are squared.

📖 Key Formula: R² (Coefficient of Determination)

R² = 1 − (SS_res / SS_tot)

R² measures what proportion of the variance in the output the model explains. R² = 1.0 means perfect prediction; R² = 0 means the model is no better than predicting the mean; negative R² means it's actively worse.

Section 5

The Iterative Nature of ML: It's Never One-and-Done

A critical mindset for any ML practitioner: the workflow is not linear. After model evaluation, you almost always return to earlier stages to improve results. This iterative cycle is what separates data scientists from people who just run a script.

Common Iteration Scenarios

Model underfits (high training error)? → Add more features, try a more complex model, train for more epochs, reduce regularisation.

Model overfits (low training error, high test error)? → Collect more data, simplify the model, add regularisation (L1/L2), use dropout for neural networks.

Wrong metric? → Re-evaluate whether accuracy is the right measure. For imbalanced datasets (e.g., 99% healthy patients), use F1-score or ROC-AUC instead.

🗞️ Analogy: Calibrating a Weighing Scale

Building an ML model is like calibrating a precision scale for a laboratory. You don't weigh one reference mass and declare it done. You weigh multiple references, adjust the calibration screw, recheck, adjust again, test against a completely independent set of reference masses, and document the uncertainty. Iteration is not failure — it is the correct scientific process.

  • Normalisation: Scaling to [0,1] or z-score
  • Data Leakage: Test data "leaking" into training (bad!)

  • Underfitting: Bad on both train and test
  • Loss Function: Objective to minimise during training

  • Regression → predicts "how much"
  • Classification → predicts "which class"
  • Regression metrics: MSE, MAE, R²
  • Classification: Accuracy, F1, ROC-AUC

5 Questions · Select & Submit

Question 01

In the ML workflow, what is the primary purpose of the test set?

Correct: C. The test set must never be seen by the model during training or tuning. It provides a final, honest estimate of real-world performance. Hyperparameter tuning uses a separate validation set. Using the test set for tuning is "data leakage" — a critical error.

Question 02

Which ML workflow stage involves creating interaction terms, encoding categorical variables, and removing irrelevant features?

Correct: D. Feature Engineering is the process of transforming raw input variables into more informative representations for the model. EDA (B) is about understanding the data; Feature Engineering (D) is about improving it as model input.

Question 03

A machine learning model achieves 98% accuracy on training data but only 62% on test data. This is a classic case of:

Correct: B. Overfitting occurs when a model learns the training data "too well" — including its noise and outliers — and fails to generalise to new data. High training accuracy + low test accuracy is the diagnostic signature of overfitting.

Question 04

Which evaluation metric is most appropriate for a regression problem predicting crop yield in kg/hectare?

Correct: A. Crop yield is a continuous numerical output → regression problem → use error-based metrics (RMSE, MAE, R²). Accuracy, F1, Precision, Recall, and ROC-AUC are designed for classification problems where the output is a category.

Question 05

Exploratory Data Analysis (EDA) is best described as:

Correct: C. EDA is a stage of discovery, not model building. You use summary statistics (mean, variance), visualisations (histograms, scatter plots), and correlation analysis to understand what's in your data before making any modeling decisions. It informs Feature Engineering (B).
